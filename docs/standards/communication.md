# コミュニケーション標準

## Purpose

この文書は、ユーザー向けの回答と primary docs で不要な略語や独自用語を増やさないための共通ルールです。

## Rules

- 略語だけで説明しない。初出では正式名または平易な日本語を先に書き、必要な場合だけ後ろに略語を添える。
- その回答や文書で 1 回しか出ない語は略さない。
- repo 独自の呼び名を作る前に、既存の一般語で言い換えられないか確認する。
- literal として扱う必要があるコマンド名、環境変数名、ファイル名、外部仕様の正式名は例外とする。
- `DESIGN.md` と `docs/design/README.md` では、この共通ルールに加えて design 文書向けの追加ルールも守る。

## Preferred Forms

| 避ける | 使う |
| --- | --- |
| 設計判断を略語だけで呼ぶ表現 | 設計判断メモ |
| `TDD` | テスト駆動開発（TDD） |
| `CI` | 継続的インテグレーション（CI） |
| `MCP` | Model Context Protocol（MCP） |
| `CLI` | コマンドラインツール（CLI） |
| `OAuth` | OAuth 認証 |

## Design Docs

`DESIGN.md`、`docs/design/README.md`、`DESIGN.sample.md` では、次を design 文書向けの追加ルールとして扱う。

- `B2B` は `企業間取引（B2B）` を書いたうえで、文書内で繰り返し使う場合だけ再利用する。
- `LP` は `ランディングページ（LP）` を書いたうえで、文書内で繰り返し使う場合だけ再利用する。
- `CTA` は `行動喚起（CTA）` を書いたうえで、文書内で繰り返し使う場合だけ再利用する。
- `UI` は `ユーザーインターフェース（UI）` を書いたうえで、文書内で繰り返し使う場合だけ再利用する。
- 1 回しか使わないなら、略語を再利用せず正式名だけで書く。

## Notes

- ユーザーが略語だけで質問した場合でも、回答側では必要に応じて展開して返す。
- docs の見出しは短くてよいが、本文では略語の意味が追えるようにする。
